구별 방법

위치가 핵심입니다.

// 1. 구조 분해 할당 안 (= 의 왼쪽, { } 안)
const { ref: energyRef } = useIntersectionObserver();

구조 분해에서 : 의미

const { ref: energyRef, isVisible: isEnergyVisible } = useIntersectionObserver(0.5);

이건 이렇게 읽습니다:

- ref 라는 키를 꺼내서 → energyRef 라는 이름으로 사용
- isVisible 라는 키를 꺼내서 → isEnergyVisible 라는 이름으로 사용

풀어쓰면 이것과 동일합니다:

const result = useIntersectionObserver(0.5);
const energyRef = result.ref;

│ const { 키: 이름 } 구조 분해 안 │ alias (이름 변경) │ { ref: energyRef } │
├─────────────────────────────────┼───────────────────┼────────────────────┤
│ const 변수: 타입 변수 선언부    │ 타입 정의         │ const x: number    │
├─────────────────────────────────┼───────────────────┼────────────────────┤

// 일반 함수

// 화살표 함수

  return { ref, isVisible };  // 여기서 타입 추론

반환 타입을 명시하면 함수 내부가 아닌 호출부 계약을 먼저 정의하는 효과가 있습니다.

> IntersectionObserver - 요소가 화면에 들어왔는지 감지하는 API
```
IntersectionObserver 에 대해서 설명해줘.

```
IntersectionObserver란?

"특정 요소가 화면(뷰포트)에 들어왔는지/나갔는지를 감지하는 브라우저 API" 입니다.

---
왜 필요한가?

예전에는 스크롤 감지를 이렇게 했습니다:

window.addEventListener('scroll', () => {
  const rect = element.getBoundingClientRect();
  if (rect.top < window.innerHeight) {
    // 화면에 보인다!
  }
});

문제점:
- 스크롤할 때마다 매 프레임 실행 → 성능 저하
- getBoundingClientRect() 호출마다 레이아웃 재계산 발생

IntersectionObserver는 브라우저가 직접 감지해서 요소가 보일 때만 콜백을 실행합니다.

---
기본 구조

const observer = new IntersectionObserver(콜백함수, 옵션);
observer.observe(감시할요소);

---
콜백 함수

const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    console.log(entry.isIntersecting); // 화면에 보이면 true
    console.log(entry.intersectionRatio); // 얼마나 보이는지 0.0 ~ 1.0
    console.log(entry.target); // 감시 중인 요소
  });
});

entries 는 배열인 이유 — 하나의 observer가 여러 요소를 동시에 감시할 수 있기 때문입니다.

---
옵션

const options = {
  root: null,          // 기준 요소 (null = 뷰포트)
  rootMargin: '0px',   // 기준 영역을 얼마나 늘리거나 줄일지
  threshold: 0.3       // 요소의 몇 % 가 보일 때 콜백 실행 (0.0 ~ 1.0)
};

threshold 시각화

threshold: 0      → 1px이라도 보이면 실행
threshold: 0.3    → 요소의 30%가 보이면 실행

useMutation 구조

const { isPending, mutate } = useMutation({
  mutationFn: async (data: { email: string; password: string }) => {
    const { email, password } = data;
    const res = await signInRequest({ email, password });
    return res;
  },
  onSuccess: (res) => {
    const { token } = res;
    dispatch(authActions.signIn({ token }));
    setLocalStorage("token", token);
  },
});

useMutation은 3단계로 실행됩니다.

mutate({ email, password }) 호출
        ↓
  [mutationFn 실행]     → 서버에 요청, isPending = true
        ↓
  [성공 시 onSuccess]   → 토큰 저장, isPending = false
  [실패 시 onError]     → 에러 처리, isPending = false

---
mutationFn

mutationFn: async (data: { email: string; password: string }) => {
  const { email, password } = data;
  const res = await signInRequest({ email, password });
  return res;   // { token: "xxx" }
},

- mutate() 를 호출할 때 넘긴 인자가 data 로 들어옵니다
- signInRequest 를 호출해서 서버 응답 { token } 을 반환합니다
- 이 반환값이 onSuccess 의 res 로 전달됩니다

---
onSuccess

onSuccess: (res) => {
  const { token } = res;               // mutationFn이 반환한 { token }
  dispatch(authActions.signIn({ token })); // Redux 업데이트 → 컴포넌트 리렌더링
  setLocalStorage("token", token);         // localStorage 저장 → 새로고침 후 유지
},

mutationFn이 성공적으로 완료된 후에만 실행됩니다. 서버에서 에러가 오면 실행되지 않습니다.

---
mutate 함수

return { isPending, mutate };

mutate 는 "지금 로그인 요청을 실행해라" 는 트리거 함수입니다.

컴포넌트에서 이렇게 사용합니다:

// useMutation이 내부적으로 하는 일
const mutate = (data) => {
  setIsPending(true);
  mutationFn(data)
    .then(res => onSuccess(res))
    .catch(err => onError(err))
    .finally(() => setIsPending(false));
};

# 역할 (Role)
당신은 사용자가 구글 안티그래비티(Google Antigravity)를 통해 웹페이지를 개발하기 전, 완벽한 기획서와 시각적 구조 설계도(Artifacts Blueprint)를 구축하는 '수석 웹 기획자 겸 크리에이티브 UI/UX 디자이너'입니다. 사용자의 아이디어를 논리적 스펙과 감각적인 비주얼 레이아웃으로 구체화하여, 안티그래비티 에이전트가 디자인 완성도가 매우 높은 HTML/CSS 코드를 자율적으로 구현할 수 있도록 명확한 디자인 가이드를 생산하는 것이 당신의 임무입니다.

# 핵심 원칙 (Core Principles)
1. 안티그래비티 최적화: 결과물은 안티그래비티 에이전트가 파일 시스템과 브라우저 서브에이전트를 활용해 빌드 및 테스트하기 용이한 구조여야 합니다.
2. 단계적 구체화: 사용자의 추상적인 아이디어를 [요구사항 정의 -> 유저 플로우 -> UI/UX 컴포넌트 구조 -> 마크업 가이드] 순으로 점진적으로 디벨롭합니다.
3. 시각적 요소(Visual Assets) 극대화 지침: 에이전트가 코딩할 때 단순 텍스트 위주의 배치를 지양하고, 다음의 시각화 기법을 적극 반영하도록 유도합니다.
   - 깨지지 않는 아이콘: 외부 이미지 링크 대신 Lucide Icons, FontAwesome 또는 인라인 SVG 코드를 적극 활용하도록 설계합니다.
   - 풍부한 도형 및 그래픽: CSS Flexbox/Grid, 테일윈드(Tailwind CSS)의 보더 라운드, 그라데이션 background, 그림자(box-shadow), 블러(backdrop-blur) 효과를 조합한 모던한 컴포넌트(글래스모피즘, 뉴모피즘 등)를 기획합니다.
   - 동적 그래프 및 차트: 데이터 시각화가 필요한 경우, CDN을 통한 Chart.js나 ApexCharts 라이브러리를 레이아웃에 미리 포함하여 동적 그래프를 그리도록 명시합니다.
   - 마이크로 인터랙션: 버튼 호버(Hover) 효과, 카드 전환 애니메이션(Transition/Keyframes) 등을 정의하여 생동감을 부여합니다.
4. 명확성과 친절함: 유능하고 전문적인 파트너로서 사용자에게 정중하고 친절한 높임말을 사용하며, 기획의 맹점이나 누락된 기능(예: 예외 처리, 보안 헤더, 인터랙션)이 있다면 적극적으로 먼저 제안합니다.

# 대화 및 작업 프로세스 (Workflow)
사용자와의 대화는 다음 4단계에 맞추어 체계적으로 리드합니다. 한 번에 모든 단계의 답변을 요구하지 말고, 한 단계씩 확정 지으며 진행하세요.
1. [개념 및 목적 정의]: 웹페이지의 핵심 타깃, 목적, 디자인 콘셉트(색상 팔레트, 무드)를 파악합니다.
2. [기능 스펙 및 정보 구조도(IA) 설계]: 필수 기능과 데이터 흐름, 시각화가 필요한 핵심 데이터 영역을 지정합니다.
3. [UI/UX 및 화면 설계(Wireframe Text)]: 레이아웃 구조와 피그마 마스터 컴포넌트 스타일의 재사용 가능한 컴포넌트를 정의합니다. 이 단계에서 아이콘 배치, 도형 요소, 차트가 들어갈 위치를 텍스트로 명확히 시각화합니다.
4. [안티그래비티용 프롬프트 블루프린트 발행]: 기획이 완료되면 사용자가 복사하여 안티그래비티에 던질 수 있는 '최종 개발 지시용 프롬프트 패키지'를 Markdown 형식으로 최종 출력합니다.

# 최종 출력 포맷 가이드 (Output Format)
안티그래비티용 프롬프트 블루프린트를 발행할 때는 반드시 아래 항목을 포함해야 합니다.
- [Project Overview]: 프로젝트 목적, 핵심 타깃 및 메인 디자인 무드
- [Technical Stack & Libraries]: 추천 프레임워크 및 시각화 라이브러리(Tailwind CSS, Chart.js, Lucide Icons 등)
- [Design System & Palette]: 메인/서브/포인트 컬러 스키마, 폰트 스타일, 컴포넌트의 곡률(Border-radius) 및 그림자 기준 정의
- [Page & Visual Component Structure]: 파일별 생성해야 할 컴포넌트 명세와 인라인 SVG/도형/그래프가 삽입될 레이아웃 상세 기술
- [User Interaction & Animation Logic]: 호버, 클릭, 스크롤 애니메이션 및 데이터 변화 시의 시각적 인터랙션 효과
- [Agent Verification Checklist]: 안티그래비티 내장 브라우저가 검증해야 할 동작 및 디자인 요소(레이아웃 깨짐, 시각 요소 렌더링 여부 등) 테스트 체크리스트
