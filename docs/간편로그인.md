## 1. 일반 로그인 vs 구글 간편 로그인 비교

### 일반 로그인 (이메일/비밀번호)

```
사용자가 직접 이메일 + 비밀번호 입력
        ↓
우리 서버에서 직접 검증
        ↓
토큰 발급
```

- 우리 서버가 비밀번호를 직접 관리
- 회원가입 → 로그인 두 단계 필요
- 비밀번호 분실 처리, 보안 관리를 우리가 직접 해야 함

### 구글 간편 로그인

```
사용자가 구글 계정으로 인증 (구글이 신원 보증)
        ↓
구글이 "이 사람 맞아요" 라는 증명서(ID Token) 발급
        ↓
우리 서버는 증명서만 확인 후 토큰 발급
```

- 비밀번호를 우리가 관리하지 않음
- 구글 계정이 있으면 별도 회원가입 불필요 (신규 유저는 자동 가입 처리)
- 보안 책임을 구글이 담당

## 2. 전체 흐름도

```
[사용자]
  │
  │ ① "구글 간편 로그인" 버튼 클릭
  ▼
[LoginScreen]
  │
  │ ② useGoogleLogin() → 구글 팝업 오픈
  ▼
[Google OAuth 서버]
  │
  │ ③ 사용자가 구글 계정 선택/로그인
  │ ④ 구글이 credential(ID Token) 발급 후 onSuccess 콜백 실행
  ▼
[LoginScreen - onSuccess 콜백]
  │
  │ ⑤ credential에서 사용자 정보 추출 (이름, 구글 고유ID)
  │ ⑥ useLogin().mutate({ sns_provider_code, sns_user_key, nickname }) 호출
  ▼
[useLogin Hook]
  │
  │ ⑦ loginRequest() 호출
  ▼
[loginApi]
  │
  │ ⑧ POST /api/login 요청
  ▼
[목서버 Handler - /api/login]
  │
  │ ⑨ sns_user_key로 기존 회원 검색
  │    - 신규면: users에 자동 추가 (회원가입 처리)
  │    - 기존이면: 해당 유저 사용
  │ ⑩ 세션 토큰 발급 후 응답 { token }
  ▼
[useLogin Hook - onSuccess]
  │
  │ ⑪ dispatch(authActions.signIn({ token })) → Redux 저장
  │ ⑫ setLocalStorage("token", token) → localStorage 저장
  ▼
[LoginScreen - mutate onSuccess]
  │
  │ ⑬ navigate("/memo") → 메인 화면 이동
  ▼
[로그인 완료]
```

## 3. 구글 OAuth 동작 원리

### ID Token이란?

구글이 발급하는 JWT(JSON Web Token)입니다. 이 안에 사용자 정보가 담겨 있습니다.

```
ID Token (JWT) 디코딩 시 내용:
{
  "sub": "1234567890",        ← 구글 고유 사용자 ID (sns_user_key로 사용)
  "name": "홍길동",
  "email": "hong@gmail.com",
  "picture": "https://...",
  "iat": 1720000000,
  "exp": 1720003600
}
```

`sub` 값이 구글 계정마다 고유하게 부여되는 ID입니다. 이메일은 변경될 수 있지만 `sub`는 변경되지 않아 사용자 식별자로 사용합니다.

### sns_provider_code란?

어떤 소셜 서비스로 로그인했는지 구분하는 코드입니다.

```
"google"  → 구글 로그인
"kakao"   → 카카오 로그인
"naver"   → 네이버 로그인
```

같은 이메일이라도 구글/카카오/네이버는 별개의 계정으로 관리합니다.

## 5. 화면 (LoginScreen)

현재 `login-screen.tsx`의 `handleGoogleLogin`은 `alert`만 호출합니다. 이를 실제 구글 OAuth 로직으로 교체합니다.

### 변경 전 (현재 코드)

```tsx
// src/screens/login-screen.tsx
const handleGoogleLogin = () => {
  alert("구글 간편 로그인으로 연결됩니다.");  // ← 미구현 상태
};
```

### 변경 후

```tsx
// src/screens/login-screen.tsx
import { useGoogleLogin } from "@react-oauth/google";
import { jwtDecode } from "jwt-decode";
import { useNavigate } from "react-router-dom";
import useLogin from "../hooks/use-login";

type GoogleCredentialPayload = {
  sub: string;    // 구글 고유 사용자 ID
  name: string;   // 사용자 이름
  email: string;
  picture: string;
};

const LoginScreen = () => {
  const navigate = useNavigate();
  const { mutate: loginMutate } = useLogin();

  const handleGoogleLogin = useGoogleLogin({
    // ① 구글 인증 성공 시 콜백
    onSuccess: (tokenResponse) => {

      // ② ID Token에서 사용자 정보 추출
      const decoded = jwtDecode<GoogleCredentialPayload>(
        tokenResponse.credential
      );

      // ③ 우리 서버에 로그인 요청
      loginMutate(
        {
          sns_provider_code: "google",
          sns_user_key: decoded.sub,    // 구글 고유 ID
          nickname: decoded.name,
        },
        {
          onSuccess: () => {
            navigate("/memo");
          },
          onError: () => {
            alert("로그인에 실패했습니다. 다시 시도해주세요.");
          },
        }
      );
    },
    onError: () => {
      alert("구글 로그인이 취소되었습니다.");
    },
    flow: "implicit",  // 팝업 방식
  });

  return (
    <div>
      {/* ... 기존 UI ... */}
      <SnsLoginButtonComponent
        type="google"
        onClick={() => handleGoogleLogin()}
      />
    </div>
  );
};
```

### 흐름 설명

```
handleGoogleLogin() 호출
        ↓
구글 팝업 오픈 (useGoogleLogin이 처리)
        ↓
사용자가 구글 계정 선택
        ↓
onSuccess({ credential: "eyJhbGc..." })
        ↓
jwtDecode(credential) → { sub, name, email, picture }
        ↓
loginMutate({ sns_provider_code: "google", sns_user_key: sub, nickname: name })
```

## 6. Hook (useLogin)

현재 프로젝트의 `use-login.ts`는 이미 구현되어 있습니다.

```ts
// src/hooks/use-login.ts (현재 코드 - 변경 불필요)
import { useDispatch } from "react-redux";
import { authActions } from "../store/auth-slice";
import { useMutation } from "@tanstack/react-query";
import { setLocalStorage } from "../utils/local-storage";
import { loginRequest } from "../api/login-api";

const useLogin = () => {
  const dispatch = useDispatch();

  const { isPending, mutate } = useMutation({
    // ① 서버에 로그인 요청
    mutationFn: async (data: {
      sns_provider_code: string;
      sns_user_key: string;
      nickname: string;
    }) => {
      const res = await loginRequest(data);
      return res;
    },
    // ② 성공 시 토큰 저장
    onSuccess: (res) => {
      const { token } = res;
      dispatch(authActions.signIn({ token }));  // Redux 업데이트 → 컴포넌트 리렌더링
      setLocalStorage("token", token);           // localStorage 저장 → 새로고침 후 유지
    },
  });

  return { isPending, mutate };
};

export default useLogin;
```

### useSignIn과의 차이점

| | `useSignIn` | `useLogin` |
|---|---|---|
| **용도** | 이메일/비밀번호 로그인 | SNS 간편 로그인 |
| **전달 데이터** | `{ email, password }` | `{ sns_provider_code, sns_user_key, nickname }` |
| **호출 API** | `signInRequest` → POST /signin | `loginRequest` → POST /login |

onSuccess 이후 동작(Redux 저장, localStorage 저장)은 동일합니다.

## 7. API (loginApi)

현재 프로젝트의 `login-api.ts`도 이미 구현되어 있습니다.

```ts
// src/api/login-api.ts (현재 코드 - 변경 불필요)
import { isAxiosError } from "axios";
import z from "zod";
import client, { ApiError } from "./client";

export const loginRequest = async (data: {
  sns_provider_code: string;
  sns_user_key: string;
  nickname: string;
}) => {
  try {
    // ① POST /api/login 요청
    const res = await client.post("/login", data);

    // ② 응답 데이터 타입 검증 (zod)
    const resDataSchema = z.object({
      token: z.string(),
    });
    const parsed = resDataSchema.safeParse(res.data);
    if (parsed.success === false) {
      throw parsed.error;
    }

    // ③ 검증된 데이터 반환
    return parsed.data;  // { token: "xxx" }

  } catch (err) {
    // ④ Axios 에러 → ApiError로 변환
    if (isAxiosError(err)) {
      throw new ApiError(err.response?.data.errorCode);
    }
    throw err;
  }
};
```

### zod 검증을 하는 이유

서버 응답이 예상한 형태인지 런타임에 확인합니다.

```ts
// 서버가 { token: 123 } 처럼 잘못된 타입을 보내도
// TypeScript만으로는 런타임에 잡을 수 없음
// zod.safeParse가 런타임에 실제 값을 검사해서 에러를 잡아줌
const resDataSchema = z.object({
  token: z.string(),  // token이 반드시 string이어야 함
});
```

## 8. 목서버 Handler

현재 `handlers.ts`의 `/api/login` 핸들러도 이미 구현되어 있습니다.

```ts
// src/server/msw/handlers.ts (현재 코드 - 변경 불필요)
http.post("/api/login", async ({ request }) => {
  const body = await request.json();
  await delay(500);  // 실제 서버 통신 시간 시뮬레이션

  const { sns_provider_code, sns_user_key, nickname } =
    body as { sns_provider_code: string; sns_user_key: string; nickname: string };

  // ① sns_user_key로 기존 회원 검색
  const foundUser = lsDatabase.users.find(
    (user) =>
      user.sns_provider_code === sns_provider_code &&
      user.sns_user_key === sns_user_key
  );

  let loginUser: User;

  // ② 신규 유저면 자동 회원가입 처리
  if (!foundUser) {
    const user_id = getNextUserId(lsDatabase.users);
    const newUser: User = {
      user_id,
      sns_provider_code,
      sns_user_key,
      nickname,
      status: "Y",
      created_at: dayjs().format("YYYY-MM-DD HH:mm:ss"),
      last_login_at: "",
    };
    lsDatabase.users.push(newUser);
    loginUser = newUser;
  } else {
    // ③ 기존 유저면 그대로 사용
    loginUser = foundUser;
  }

  // ④ 세션 토큰 발급 (기존 세션 제거 후 새로 생성)
  const token = crypto.randomUUID();
  lsDatabase.sessions = lsDatabase.sessions.filter(
    (s) => s.user_id !== loginUser.user_id
  );
  lsDatabase.sessions.push({ token, user_id: loginUser.user_id });

  // ⑤ 변경된 DB 저장
  saveDatabase(lsDatabase);

  // ⑥ 토큰 응답
  return HttpResponse.json({ token });
}),
```

### 신규 유저 자동 가입 처리 흐름

```
sns_user_key로 users 테이블 검색
        ↓
┌───────────────────┬─────────────────────────┐
│ 기존 회원         │ 신규 회원               │
│ foundUser 있음    │ foundUser 없음          │
│                   │                         │
│ 그대로 사용       │ users에 신규 추가       │
└───────────────────┴─────────────────────────┘
        ↓
세션 토큰 발급 (crypto.randomUUID())
        ↓
{ token } 응답
```

이메일/비밀번호 로그인과 달리 **회원가입 단계가 없습니다.** 처음 로그인하는 유저는 자동으로 가입됩니다.

## 9. 레이어별 역할 정리

```
[LoginScreen]
  - "구글 간편 로그인" 버튼 렌더링
  - useGoogleLogin()으로 구글 팝업 실행
  - onSuccess에서 ID Token 디코딩 후 useLogin().mutate() 호출
  - 로그인 완료 후 페이지 이동
        ↓
[useLogin Hook]
  - useMutation으로 비동기 상태 관리 (isPending 등)
  - loginRequest() 호출
  - 성공 시 Redux + localStorage에 token 저장
        ↓
[loginRequest (login-api.ts)]
  - POST /api/login HTTP 요청
  - zod로 응답 타입 검증
  - Axios 에러 → ApiError 변환
        ↓
[MSW Handler - /api/login]
  - sns_user_key로 기존 회원 여부 확인
  - 신규면 자동 회원가입 처리
  - 세션 토큰 발급 후 { token } 응답
```

### 각 레이어가 아는 것 / 모르는 것

| 레이어 | 아는 것 | 모르는 것 |
|---|---|---|
| `LoginScreen` | 구글 팝업, 사용자 정보, 화면 이동 | HTTP 요청 방법 |
| `useLogin` | 비동기 상태, Redux, localStorage | 구글 팝업, HTTP 세부사항 |
| `loginRequest` | HTTP 요청, 응답 검증 | React, Redux, 구글 |
| MSW Handler | DB 조회/저장, 토큰 발급 | 클라이언트 구현 |

각 레이어가 자신의 역할만 알고 나머지는 모르는 구조입니다. 이를 **관심사의 분리(Separation of Concerns)** 라고 하며, 한 레이어를 변경해도 다른 레이어에 영향을 주지 않습니다.

> **실제 서버 연동 시 변경 필요한 부분**
> - MSW Handler → 실제 서버 엔드포인트로 교체
> - 실제 서버에서는 구글 ID Token을 구글 서버에 검증 요청 후 `sub` 추출
> - `login-api.ts`, `use-login.ts`, `LoginScreen`은 변경 불필요

## 10. 실제 서버 - Google ID Token 검증

### 왜 서버에서 검증해야 하는가?

클라이언트(프론트엔드)에서 `jwtDecode`로 ID Token을 디코딩하는 것은 **내용을 읽는 것**일 뿐,
**진짜 구글이 발급한 토큰인지 위변조 여부는 확인하지 않습니다.**

```
jwtDecode("eyJhbGc...")
  → 그냥 base64 디코딩만 함
  → 누가 만든 토큰인지, 유효한 토큰인지 검증 안 함
  → 공격자가 임의로 만든 토큰도 통과됨 ← 보안 취약점
```

실제 서버에서는 구글에 "이 토큰 진짜 맞아?" 라고 확인 요청을 해야 합니다.

```
[클라이언트]                [우리 서버]              [구글 서버]
     │                          │                        │
     │  POST /api/login         │                        │
     │  { id_token: "eyJ..." }  │                        │
     │ ─────────────────────── ▶│                        │
     │                          │  GET tokeninfo?id_token=eyJ...
     │                          │ ──────────────────────▶│
     │                          │                        │
     │                          │  { sub, email, name }  │
     │                          │ ◀──────────────────────│
     │                          │                        │
     │                          │ sub로 회원 조회/가입   │
     │                          │ 우리 서비스 token 발급 │
     │  { token }               │                        │
     │ ◀─────────────────────── │                        │
```

### 클라이언트 코드 변경

서버 검증 방식에서는 클라이언트가 ID Token을 직접 디코딩하지 않고,
**구글에서 받은 ID Token 원본을 그대로 서버로 전달**합니다.

```tsx
// src/screens/login-screen.tsx 변경
const handleGoogleLogin = useGoogleLogin({
  onSuccess: (tokenResponse) => {

    // jwtDecode 제거 - 서버에서 검증하므로 클라이언트에서 디코딩 불필요
    loginMutate(
      {
        sns_provider_code: "google",
        id_token: tokenResponse.credential, // ID Token 원본을 그대로 전달
      },
      {
        onSuccess: () => navigate("/memo"),
        onError: () => alert("로그인에 실패했습니다."),
      }
    );
  },
  flow: "implicit",
});
```

```ts
// src/api/login-api.ts 변경
export const loginRequest = async (data: {
  sns_provider_code: string;
```

### 서버 구현 (Node.js + Express 예시)

#### 방법 1: 구글 tokeninfo 엔드포인트 사용 (간단)

구글이 제공하는 검증 API에 직접 요청하는 방식입니다. 별도 라이브러리가 필요 없습니다.

```js
// server/routes/auth.js (Node.js + Express)
const express = require("express");
const axios = require("axios");
const router = express.Router();

router.post("/login", async (req, res) => {
  const { sns_provider_code, id_token } = req.body;

  if (sns_provider_code === "google") {

    // ① 구글 tokeninfo API에 ID Token 검증 요청
    let googleUser;
    try {
      const response = await axios.get(
        `https://oauth2.googleapis.com/tokeninfo?id_token=${id_token}`
      );
      googleUser = response.data;
      // googleUser = {
      //   sub: "1234567890",        ← 구글 고유 ID
      //   email: "hong@gmail.com",
      //   name: "홍길동",
      //   aud: "우리앱-client-id.apps.googleusercontent.com",
      //   exp: "1720003600",
      //   ...
      // }
    } catch (err) {
      // 토큰이 유효하지 않으면 구글이 400 에러 반환
      return res.status(401).json({ errorCode: "INVALID_GOOGLE_TOKEN" });
    }

    // ② aud(audience) 검증 - 우리 앱의 Client ID인지 확인
    //    다른 앱의 토큰으로 로그인 시도하는 것을 막음
    const GOOGLE_CLIENT_ID = process.env.GOOGLE_CLIENT_ID;
    if (googleUser.aud !== GOOGLE_CLIENT_ID) {
      return res.status(401).json({ errorCode: "INVALID_GOOGLE_TOKEN" });
    }

    // ③ 만료 시간 검증
    const now = Math.floor(Date.now() / 1000);
    if (parseInt(googleUser.exp) < now) {
      return res.status(401).json({ errorCode: "EXPIRED_GOOGLE_TOKEN" });
    }

    // ④ sub(구글 고유 ID)로 기존 회원 조회
    const sns_user_key = googleUser.sub;
    let user = await db.users.findOne({ sns_provider_code: "google", sns_user_key });

    // ⑤ 신규 유저면 자동 회원가입
    if (!user) {
      user = await db.users.create({
        sns_provider_code: "google",
        sns_user_key,
        nickname: googleUser.name,
        status: "Y",
        created_at: new Date(),
      });
    }

    // ⑥ 우리 서비스 토큰 발급
    const token = generateSessionToken(); // crypto.randomUUID() 등
    await db.sessions.create({ token, user_id: user.user_id });

    return res.json({ token });
  }

  return res.status(400).json({ errorCode: "UNSUPPORTED_PROVIDER" });
});
```

#### 방법 2: google-auth-library 사용 (권장)

구글 공식 라이브러리를 사용하면 서명 검증, 만료 시간 확인, aud 검증을 자동으로 처리합니다.

```bash
npm install google-auth-library
```

```js
// server/routes/auth.js
const { OAuth2Client } = require("google-auth-library");

const GOOGLE_CLIENT_ID = process.env.GOOGLE_CLIENT_ID;
const client = new OAuth2Client(GOOGLE_CLIENT_ID);

router.post("/login", async (req, res) => {
  const { sns_provider_code, id_token } = req.body;

  if (sns_provider_code === "google") {

    // ① google-auth-library로 ID Token 검증 (서명 + 만료 + aud 모두 자동 확인)
    let payload;
    try {
      const ticket = await client.verifyIdToken({
        idToken: id_token,
        audience: GOOGLE_CLIENT_ID,
      });
      payload = ticket.getPayload();
      // payload = {
      //   sub: "1234567890",
      //   email: "hong@gmail.com",
      //   name: "홍길동",
      //   picture: "https://...",
      //   email_verified: true,
      // }
    } catch (err) {
      // 위변조, 만료, aud 불일치 시 자동으로 에러 throw
      return res.status(401).json({ errorCode: "INVALID_GOOGLE_TOKEN" });
    }

    // ② 이메일 인증 여부 확인 (선택)
    if (!payload.email_verified) {
      return res.status(401).json({ errorCode: "EMAIL_NOT_VERIFIED" });
    }

    const sns_user_key = payload.sub; // 구글 고유 ID

    // ③ 기존 회원 조회 → 없으면 자동 가입
    let user = await db.users.findOne({ sns_provider_code: "google", sns_user_key });
    if (!user) {
      user = await db.users.create({
        sns_provider_code: "google",
        sns_user_key,
        nickname: payload.name,
        status: "Y",
        created_at: new Date(),
      });
    }

    // ④ 우리 서비스 토큰 발급
    const token = crypto.randomUUID();
    await db.sessions.create({ token, user_id: user.user_id });

    return res.json({ token });
  }
});
```

### 두 방법 비교

| | tokeninfo API | google-auth-library |
|---|---|---|
| **설치** | 불필요 | `npm install google-auth-library` |
| **검증 방식** | 구글 서버에 HTTP 요청 | 로컬에서 서명 직접 검증 |
| **속도** | 느림 (네트워크 왕복 발생) | 빠름 (로컬 처리) |
| **서명 검증** | 구글 서버가 처리 | 라이브러리가 직접 처리 |
| **권장 여부** | 간단한 테스트용 | 실서비스 권장 |

### aud(audience) 검증이 중요한 이유

```
공격 시나리오:
  공격자가 다른 앱(게임앱)에서 구글 로그인 후 ID Token 획득
        ↓
  그 토큰을 우리 서버 /api/login에 전송
        ↓
  aud 검증 없으면 → 다른 사람 계정으로 우리 서비스 로그인 가능 ← 보안 취약점

aud 검증:
  ID Token 안의 aud 값 = 게임앱 Client ID
  우리 서버의 GOOGLE_CLIENT_ID = 우리앱 Client ID
  → 불일치 → 401 에러 반환 → 공격 차단
```

## 11. Google Cloud Console 클라이언트 ID 발급

### 전체 순서

```
1. Google Cloud Console 프로젝트 생성
2. OAuth 동의 화면 설정
3. OAuth 클라이언트 ID 생성
4. 발급된 클라이언트 ID를 .env에 등록
```

---

### 1단계: Google Cloud Console 접속 및 프로젝트 생성

1. `console.cloud.google.com` 접속 (구글 계정 로그인 필요)
2. 상단 프로젝트 선택 드롭다운 클릭
3. **"새 프로젝트"** 클릭
4. 프로젝트 이름 입력 (예: `연애판별소`) → **"만들기"** 클릭
5. 생성된 프로젝트 선택

---

### 2단계: OAuth 동의 화면 설정

> 사용자가 구글 로그인 팝업에서 보는 화면을 설정합니다.

1. 왼쪽 메뉴 → **"API 및 서비스"** → **"OAuth 동의 화면"** 클릭
2. User Type 선택
   - **외부** 선택 (일반 사용자 대상 서비스) → **"만들기"**
3. 앱 정보 입력

   | 항목 | 입력값 예시 |
   |---|---|
   | 앱 이름 | 연애판별소 |
   | 사용자 지원 이메일 | 내 구글 이메일 |
   | 개발자 연락처 이메일 | 내 구글 이메일 |

4. **"저장 후 계속"** 클릭
5. 범위(Scopes) 화면 → 기본값 유지 → **"저장 후 계속"**
6. 테스트 사용자 화면 → 개발 중에는 테스트할 구글 계정 이메일 추가
7. **"저장 후 계속"** → 완료

---

### 3단계: OAuth 클라이언트 ID 생성

1. 왼쪽 메뉴 → **"API 및 서비스"** → **"사용자 인증 정보"** 클릭
2. 상단 **"+ 사용자 인증 정보 만들기"** → **"OAuth 클라이언트 ID"** 클릭
3. 애플리케이션 유형 → **"웹 애플리케이션"** 선택
4. 이름 입력 (예: `연애판별소 Web Client`)
5. **승인된 JavaScript 원본** 추가

   ```
   개발 환경:  http://localhost:5173
   운영 환경:  https://www.yourdomain.com
   ```

   > `@react-oauth/google`은 팝업 방식으로 동작하므로 JavaScript 원본만 설정하면 됩니다.
   > 리다이렉트 URI는 팝업 방식에서 불필요합니다.

6. **"만들기"** 클릭
7. 팝업에서 클라이언트 ID 확인 및 복사

   ```
   클라이언트 ID: 123456789000-abcdefghijklmnop.apps.googleusercontent.com
   ```

---

| 오류 메시지 | 원인 | 해결 방법 |
|---|---|---|
| `idpiframe_initialization_failed` | 승인된 JavaScript 원본 미등록 | Cloud Console에서 `http://localhost:5173` 추가 |

### 개발 → 운영 전환 시 주의사항

OAuth 동의 화면이 **"테스트"** 상태이면 등록된 테스트 사용자만 로그인 가능합니다.
실서비스 출시 전 **"앱 게시"** 를 통해 게시 상태로 전환해야 합니다.

```
Google Cloud Console
→ OAuth 동의 화면
→ "앱 게시" 버튼 클릭
→ 확인
```

게시 후에는 구글 검토 과정이 있을 수 있으며 (민감한 scope 요청 시),
기본 scope(`email`, `profile`, `openid`)만 사용하면 검토 없이 즉시 게시됩니다.
